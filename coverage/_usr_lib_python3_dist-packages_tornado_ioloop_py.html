16"""An I/O event loop for non-blocking sockets. 

17 

18On Python 3, `.IOLoop` is a wrapper around the `asyncio` event loop. 

19 

20Typical applications will use a single `IOLoop` object, accessed via 

21`IOLoop.current` class method. The `IOLoop.start` method (or 

22equivalently, `asyncio.AbstractEventLoop.run_forever`) should usually 

23be called at the end of the ``main()`` function. Atypical applications 

24may use more than one `IOLoop`, such as one `IOLoop` per thread, or 

25per `unittest` case. 

26 

27In addition to I/O events, the `IOLoop` can also schedule time-based 

28events. `IOLoop.add_timeout` is a non-blocking alternative to 

29`time.sleep`. 

30 

31"""

86 """A level-triggered I/O loop. 

87 

88 On Python 3, `IOLoop` is a wrapper around the `asyncio` event 

89 loop. On Python 2, it uses ``epoll`` (Linux) or ``kqueue`` (BSD 

90 and Mac OS X) if they are available, or else we fall back on 

91 select(). If you are implementing a system that needs to handle 

92 thousands of simultaneous connections, you should use a system 

93 that supports either ``epoll`` or ``kqueue``. 

94 

95 Example usage for a simple TCP server:

108 message = await stream.read_until_close() 

109 print("message from client:", message.decode().strip())

111 def connection_ready(sock, fd, events): 

112 while True: 

113 try: 

114 connection, address = sock.accept() 

115 except socket.error as e: 

116 if e.args[0] not in (errno.EWOULDBLOCK, errno.EAGAIN): 

117 raise 

118 return

121 io_loop.spawn_callback(handle_connection, connection, address)

138 By default, a newly-constructed `IOLoop` becomes the thread's current 

139 `IOLoop`, unless there already is a current `IOLoop`. This behavior 

140 can be controlled with the ``make_current`` argument to the `IOLoop` 

141 constructor: if ``make_current=True``, the new `IOLoop` will always 

142 try to become current and it raises an error if there is already a 

143 current instance. If ``make_current=False``, the new `IOLoop` will 

144 not try to become current. 

145 

146 In general, an `IOLoop` cannot survive a fork or be shared across 

147 processes in any way. When multiple processes are being used, each 

148 process should create its own `IOLoop`, which also implies that 

149 any objects which depend on the `IOLoop` (such as 

150 `.AsyncHTTPClient`) must also be created in the child processes. 

151 As a guideline, anything that starts processes (including the 

152 `tornado.process` and `multiprocessing` modules) should do so as 

153 early as possible, ideally the first thing the application does 

154 after loading its configuration in ``main()``. 

155

513 def run_sync(self, func, timeout=None): 

514 """Starts the `IOLoop`, runs the given function, and stops the loop. 

515 

516 The function must return either an awaitable object or 

517 ``None``. If the function returns an awaitable object, the 

518 `IOLoop` will run until the awaitable is resolved (and 

519 `run_sync()` will return the awaitable's result). If it raises 

520 an exception, the `IOLoop` will stop and the exception will be 

521 re-raised to the caller. 

522 

523 The keyword-only argument ``timeout`` may be used to set 

524 a maximum duration for the function. If the timeout expires, 

525 a `tornado.util.TimeoutError` is raised. 

526 

527 This method is useful to allow asynchronous calls in a 

528 ``main()`` function:: 

529 

530 async def main(): 

531 # do stuff... 

532 

533 if __name__ == '__main__': 

534 IOLoop.current().run_sync(main) 

535 

536 .. versionchanged:: 4.3 

537 Returning a non-``None``, non-awaitable value is now an error. 

538 

539 .. versionchanged:: 5.0 

540 If a timeout occurs, the ``func`` coroutine will be cancelled. 

541 

542 """ 

543 future_cell = [None] 

544 

545 def run(): 

546 try: 

547 result = func() 

548 if result is not None: 

549 from tornado.gen import convert_yielded 

550 result = convert_yielded(result) 

551 except Exception: 

552 future_cell[0] = Future() 

553 future_set_exc_info(future_cell[0], sys.exc_info()) 

554 else: 

555 if is_future(result): 

556 future_cell[0] = result 

557 else: 

558 future_cell[0] = Future() 

559 future_cell[0].set_result(result) 

560 self.add_future(future_cell[0], lambda future: self.stop()) 

561 self.add_callback(run) 

562 if timeout is not None: 

563 def timeout_callback(): 

564 # If we can cancel the future, do so and wait on it. If not, 

565 # Just stop the loop and return with the task still pending. 

566 # (If we neither cancel nor wait for the task, a warning 

567 # will be logged). 

568 if not future_cell[0].cancel(): 

569 self.stop() 

570 timeout_handle = self.add_timeout(self.time() + timeout, timeout_callback) 

571 self.start() 

572 if timeout is not None: 

573 self.remove_timeout(timeout_handle) 

574 if future_cell[0].cancelled() or not future_cell[0].done(): 

575 raise TimeoutError('Operation timed out after %s seconds' % timeout) 

576 return future_cell[0].result()

593 def add_timeout(self, deadline, callback, *args, **kwargs): 

594 """Runs the ``callback`` at the time ``deadline`` from the I/O loop. 

595 

596 Returns an opaque handle that may be passed to 

597 `remove_timeout` to cancel. 

598 

599 ``deadline`` may be a number denoting a time (on the same 

600 scale as `IOLoop.time`, normally `time.time`), or a 

601 `datetime.timedelta` object for a deadline relative to the 

602 current time. Since Tornado 4.0, `call_later` is a more 

603 convenient alternative for the relative case since it does not 

604 require a timedelta object. 

605 

606 Note that it is not safe to call `add_timeout` from other threads. 

607 Instead, you must use `add_callback` to transfer control to the 

608 `IOLoop`'s thread, and then call `add_timeout` from there. 

609 

610 Subclasses of IOLoop must implement either `add_timeout` or 

611 `call_at`; the default implementations of each will call 

612 the other. `call_at` is usually easier to implement, but 

613 subclasses that wish to maintain compatibility with Tornado 

614 versions prior to 4.0 must use `add_timeout` instead. 

615 

616 .. versionchanged:: 4.0 

617 Now passes through ``*args`` and ``**kwargs`` to the callback. 

618 """ 

619 if isinstance(deadline, numbers.Real): 

620 return self.call_at(deadline, callback, *args, **kwargs) 

621 elif isinstance(deadline, datetime.timedelta): 

622 return self.call_at(self.time() + timedelta_to_seconds(deadline), 

623 callback, *args, **kwargs) 

624 else: 

625 raise TypeError("Unsupported deadline %r" % deadline)

935 def start(self): 

936 if self._running: 

937 raise RuntimeError("IOLoop is already running") 

938 if os.getpid() != self._pid: 

939 raise RuntimeError("Cannot share PollIOLoops across processes") 

940 self._setup_logging() 

941 if self._stopped: 

942 self._stopped = False 

943 return 

944 old_current = IOLoop.current(instance=False) 

945 if old_current is not self: 

946 self.make_current() 

947 self._thread_ident = thread.get_ident() 

948 self._running = True 

949 

950 # signal.set_wakeup_fd closes a race condition in event loops: 

951 # a signal may arrive at the beginning of select/poll/etc 

952 # before it goes into its interruptible sleep, so the signal 

953 # will be consumed without waking the select. The solution is 

954 # for the (C, synchronous) signal handler to write to a pipe, 

955 # which will then be seen by select. 

956 # 

957 # In python's signal handling semantics, this only matters on the 

958 # main thread (fortunately, set_wakeup_fd only works on the main 

959 # thread and will raise a ValueError otherwise). 

960 # 

961 # If someone has already set a wakeup fd, we don't want to 

962 # disturb it. This is an issue for twisted, which does its 

963 # SIGCHLD processing in response to its own wakeup fd being 

964 # written to. As long as the wakeup fd is registered on the IOLoop, 

965 # the loop will still wake up and everything should work. 

966 old_wakeup_fd = None 

967 if hasattr(signal, 'set_wakeup_fd') and os.name == 'posix': 

968 # requires python 2.6+, unix. set_wakeup_fd exists but crashes 

969 # the python process on windows. 

970 try: 

971 old_wakeup_fd = signal.set_wakeup_fd(self._waker.write_fileno()) 

972 if old_wakeup_fd != -1: 

973 # Already set, restore previous value. This is a little racy, 

974 # but there's no clean get_wakeup_fd and in real use the 

975 # IOLoop is just started once at the beginning. 

976 signal.set_wakeup_fd(old_wakeup_fd) 

977 old_wakeup_fd = None 

978 except ValueError: 

979 # Non-main thread, or the previous value of wakeup_fd 

980 # is no longer valid. 

981 old_wakeup_fd = None 

982 

983 try: 

984 while True: 

985 # Prevent IO event starvation by delaying new callbacks 

986 # to the next iteration of the event loop. 

987 ncallbacks = len(self._callbacks) 

988 

989 # Add any timeouts that have come due to the callback list. 

990 # Do not run anything until we have determined which ones 

991 # are ready, so timeouts that call add_timeout cannot 

992 # schedule anything in this iteration. 

993 due_timeouts = [] 

994 if self._timeouts: 

995 now = self.time() 

996 while self._timeouts: 

997 if self._timeouts[0].callback is None: 

998 # The timeout was cancelled. Note that the 

999 # cancellation check is repeated below for timeouts 

1000 # that are cancelled by another timeout or callback. 

1001 heapq.heappop(self._timeouts) 

1002 self._cancellations -= 1 

1003 elif self._timeouts[0].deadline <= now: 

1004 due_timeouts.append(heapq.heappop(self._timeouts)) 

1005 else: 

1006 break 

1007 if (self._cancellations > 512 and 

1008 self._cancellations > (len(self._timeouts) >> 1)): 

1009 # Clean up the timeout queue when it gets large and it's 

1010 # more than half cancellations. 

1011 self._cancellations = 0 

1012 self._timeouts = [x for x in self._timeouts 

1013 if x.callback is not None] 

1014 heapq.heapify(self._timeouts) 

1015 

1016 for i in range(ncallbacks): 

1017 self._run_callback(self._callbacks.popleft()) 

1018 for timeout in due_timeouts: 

1019 if timeout.callback is not None: 

1020 self._run_callback(timeout.callback) 

1021 # Closures may be holding on to a lot of memory, so allow 

1022 # them to be freed before we go into our poll wait. 

1023 due_timeouts = timeout = None 

1024 

1025 if self._callbacks: 

1026 # If any callbacks or timeouts called add_callback, 

1027 # we don't want to wait in poll() before we run them. 

1028 poll_timeout = 0.0 

1029 elif self._timeouts: 

1030 # If there are any timeouts, schedule the first one. 

1031 # Use self.time() instead of 'now' to account for time 

1032 # spent running callbacks. 

1033 poll_timeout = self._timeouts[0].deadline - self.time() 

1034 poll_timeout = max(0, min(poll_timeout, _POLL_TIMEOUT)) 

1035 else: 

1036 # No timeouts and no callbacks, so use the default. 

1037 poll_timeout = _POLL_TIMEOUT 

1038 

1039 if not self._running: 

1040 break 

1041 

1042 if self._blocking_signal_threshold is not None: 

1043 # clear alarm so it doesn't fire while poll is waiting for 

1044 # events. 

1045 signal.setitimer(signal.ITIMER_REAL, 0, 0) 

1046 

1047 try: 

1048 event_pairs = self._impl.poll(poll_timeout) 

1049 except Exception as e: 

1050 # Depending on python version and IOLoop implementation, 

1051 # different exception types may be thrown and there are 

1052 # two ways EINTR might be signaled: 

1053 # * e.errno == errno.EINTR 

1054 # * e.args is like (errno.EINTR, 'Interrupted system call') 

1055 if errno_from_exception(e) == errno.EINTR: 

1056 continue 

1057 else: 

1058 raise 

1059 

1060 if self._blocking_signal_threshold is not None: 

1061 signal.setitimer(signal.ITIMER_REAL, 

1062 self._blocking_signal_threshold, 0) 

1063 

1064 # Pop one fd at a time from the set of pending fds and run 

1065 # its handler. Since that handler may perform actions on 

1066 # other file descriptors, there may be reentrant calls to 

1067 # this IOLoop that modify self._events 

1068 self._events.update(event_pairs) 

1069 while self._events: 

1070 fd, events = self._events.popitem() 

1071 try: 

1072 fd_obj, handler_func = self._handlers[fd] 

1073 handler_func(fd_obj, events) 

1074 except (OSError, IOError) as e: 

1075 if errno_from_exception(e) == errno.EPIPE: 

1076 # Happens when the client closes the connection 

1077 pass 

1078 else: 

1079 self.handle_callback_exception(self._handlers.get(fd)) 

1080 except Exception: 

1081 self.handle_callback_exception(self._handlers.get(fd)) 

1082 fd_obj = handler_func = None 

1083 

1084 finally: 

1085 # reset the stopped flag so another start/stop pair can be issued 

1086 self._stopped = False 

1087 if self._blocking_signal_threshold is not None: 

1088 signal.setitimer(signal.ITIMER_REAL, 0, 0) 

1089 if old_current is None: 

1090 IOLoop.clear_current() 

1091 elif old_current is not self: 

1092 old_current.make_current() 

1093 if old_wakeup_fd is not None: 

1094 signal.set_wakeup_fd(old_wakeup_fd) 

1095 

1096 def stop(self): 

1097 self._running = False 

1098 self._stopped = True 

1099 self._waker.wake()

1148 def __init__(self, deadline, callback, io_loop):

1150 raise TypeError("Unsupported deadline %r" % deadline)

1155 # Comparison methods to sort by deadline, with object id as a tiebreaker

1158 # use __lt__).

1166class PeriodicCallback(object): 

1167 """Schedules the given callback to be called periodically. 

1168 

1169 The callback is called every ``callback_time`` milliseconds. 

1170 Note that the timeout is given in milliseconds, while most other 

1171 time-related functions in Tornado use seconds. 

1172 

1173 If ``jitter`` is specified, each callback time will be randomly selected 

1174 within a window of ``jitter * callback_time`` milliseconds. 

1175 Jitter can be used to reduce alignment of events with similar periods. 

1176 A jitter of 0.1 means allowing a 10% variation in callback time. 

1177 The window is centered on ``callback_time`` so the total number of calls 

1178 within a given interval should not be significantly affected by adding 

1179 jitter. 

1180 

1181 If the callback runs for longer than ``callback_time`` milliseconds, 

1182 subsequent invocations will be skipped to get back on schedule. 

1183 

1184 `start` must be called after the `PeriodicCallback` is created. 

1185 

1186 .. versionchanged:: 5.0 

1187 The ``io_loop`` argument (deprecated since version 4.1) has been removed. 

1188 

1189 .. versionchanged:: 5.1 

1190 The ``jitter`` argument is added. 

1191 """ 

1192 def __init__(self, callback, callback_time, jitter=0): 

1193 self.callback = callback 

1194 if callback_time <= 0: 

1195 raise ValueError("Periodic callback must have a positive callback_time") 

1196 self.callback_time = callback_time 

1197 self.jitter = jitter 

1198 self._running = False 

1199 self._timeout = None 

1200 

1201 def start(self): 

1202 """Starts the timer.""" 

1203 # Looking up the IOLoop here allows to first instantiate the 

1204 # PeriodicCallback in another thread, then start it using 

1205 # IOLoop.add_callback(). 

1206 self.io_loop = IOLoop.current() 

1207 self._running = True 

1208 self._next_timeout = self.io_loop.time() 

1209 self._schedule_next() 

1210 

1211 def stop(self): 

1212 """Stops the timer.""" 

1213 self._running = False 

1214 if self._timeout is not None: 

1215 self.io_loop.remove_timeout(self._timeout) 

1216 self._timeout = None 

1217 

1218 def is_running(self): 

1219 """Return True if this `.PeriodicCallback` has been started. 

1220 

1221 .. versionadded:: 4.1 

1222 """ 

1223 return self._running 

1224 

1225 def _run(self): 

1226 if not self._running: 

1227 return 

1228 try: 

1229 return self.callback() 

1230 except Exception: 

1231 self.io_loop.handle_callback_exception(self.callback) 

1232 finally: 

1233 self._schedule_next() 

1234 

1235 def _schedule_next(self): 

1236 if self._running: 

1237 self._update_next(self.io_loop.time()) 

1238 self._timeout = self.io_loop.add_timeout(self._next_timeout, self._run) 

1239 

1240 def _update_next(self, current_time): 

1241 callback_time_sec = self.callback_time / 1000.0 

1242 if self.jitter: 

1243 # apply jitter fraction 

1244 callback_time_sec *= 1 + (self.jitter * (random.random() - 0.5)) 

1245 if self._next_timeout <= current_time: 

1246 # The period should be measured from the start of one call 

1247 # to the start of the next. If one call takes too long, 

1248 # skip cycles to get back to a multiple of the original 

1249 # schedule. 

1250 self._next_timeout += (math.floor((current_time - self._next_timeout) / 

1251 callback_time_sec) + 1) * callback_time_sec 

1252 else: 

1253 # If the clock moved backwards, ensure we advance the next 

1254 # timeout instead of recomputing the same value again. 

1255 # This may result in long gaps between callbacks if the 

1256 # clock jumps backwards by a lot, but the far more common 

1257 # scenario is a small NTP adjustment that should just be 

1258 # ignored. 

1259 # 

1260 # Note that on some systems if time.time() runs slower 

1261 # than time.monotonic() (most common on windows), we 

1262 # effectively experience a small backwards time jump on 

1263 # every iteration because PeriodicCallback uses 

1264 # time.time() while asyncio schedules callbacks using 

1265 # time.monotonic(). 

1266 # https://github.com/tornadoweb/tornado/issues/2333 

1267 self._next_timeout += callback_time_sec
